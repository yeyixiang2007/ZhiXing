## 一、产品概览
本产品是一款**基于Java语言开发、以图数据结构为核心、融合最短路径算法**的校园智能路径规划导航系统，专为大学校园场景打造。
系统将校园地理信息抽象为「顶点-边」的图模型，以校园建筑/地点为顶点、通行道路为带权边，通过经典最短路径算法，为用户提供精准的校园内路径查询、导航指引、地图动态管理能力，解决校园内找路难、通用导航精度不足、临时管制绕行不及时等核心痛点。
产品核心形态分为**Java控制台基础版**（课程大作业核心交付版本）与**JavaFX/Swing GUI进阶版**（可选扩展版本），完整体现面向对象设计思想、数据结构与算法的工程化落地。

## 二、需求背景
1.  **校园场景核心痛点**：国内高校校园普遍面积大、建筑密集、内部道路复杂，新生及校外访客难以快速熟悉校园环境，常出现找不到教学楼、会议室、宿舍的情况，通用导航软件（高德、百度）对校园内部小路、楼宇内部通道、临时管制路段的覆盖度不足，无法满足校园内精细化导航需求。
2.  **用户个性化需求缺失**：在校师生存在多样化的通行需求，如上课前找最短时间到教室的路径、找途经食堂的顺路路线、避开施工/人流高峰路段的路线，通用导航无法适配校园专属场景。
3.  **课程设计适配性**：本项目完美契合Java面向对象课程大作业要求，规避烂大街的CRUD管理系统，深度融合封装、继承、多态、接口等OOP核心特性，同时覆盖图数据结构、最短路径算法、文件持久化等核心知识点，可充分体现开发人员的设计与编码能力。

## 四、产品核心目标
### 4.1 业务目标
1.  覆盖校园95%以上的核心地点（教学楼、宿舍、食堂、图书馆、校门、体育馆、行政楼等），地图数据可灵活扩展；
2.  路径计算准确率100%，无逻辑错误，可适配禁行路段的动态绕行计算；
3.  完整体现Java面向对象设计思想，代码模块化、可扩展、易维护，符合课程大作业的评分标准。

### 4.2 体验目标
1.  核心操作路径极简，用户3步内即可完成路径查询；
2.  导航指引清晰易懂，支持文字化分步导航，无专业门槛；
3.  异常场景友好提示，无崩溃、无逻辑卡死，输入容错性强。

## 六、核心业务流程
### 6.1 主流程：用户路径查询流程
1.  用户进入系统，选择「普通用户」角色，进入路径查询菜单；
2.  系统展示全量校园地点列表，用户选择/输入起点、终点；
3.  系统校验起点、终点的合法性，若不存在，提示用户重新输入；
4.  系统校验路径可达性，若无可达路径，提示不可达原因，流程结束；
5.  系统调用路径规划算法，计算最优路径，生成导航指引；
6.  系统向用户展示路径详情、分步导航指引；
7.  用户可选择重新查询，或退出系统，流程结束。

### 6.2 管理流程：管理员地图数据更新流程
1.  管理员进入系统，输入账号密码完成登录校验；
2.  校验通过，进入管理员专属菜单，选择地图数据管理功能；
3.  管理员选择操作类型：新增/修改/删除地点、新增/修改/删除道路、设置禁行路段；
4.  管理员录入对应数据，系统校验数据合法性（如地点编号不重复、道路起点终点存在）；
5.  校验通过，更新内存中的地图数据，同步写入本地持久化文件；
6.  系统提示操作成功，管理员可继续操作，或退出登录，流程结束。

### 6.3 异常流程处理
1.  **输入非法数据**：用户输入非数字、不存在的地点编号时，系统给出友好提示，引导用户重新输入，不崩溃、不退出；
2.  **路径不可达**：起点与终点之间无任何可通行路径时，系统明确提示不可达原因，而非卡死或输出错误结果；
3.  **地图文件加载失败**：系统启动时，若本地地图文件损坏/不存在，自动初始化默认校园地图，保证系统可正常运行；
4.  **权限越界**：普通用户尝试访问管理员功能时，系统拦截并提示权限不足，引导切换角色。

## 七、非功能需求
### 7.1 性能需求
1.  路径计算响应时间：单条路径计算耗时≤1s（顶点数≤200、边数≤500的场景下）；
2.  数据加载性能：系统启动时，地图数据加载耗时≤2s；
3.  并发能力：控制台版本支持单用户串行操作，GUI进阶版支持多用户同时查询路径，无数据冲突。

### 7.2 兼容性需求
1.  控制台版本：兼容JDK8及以上版本，跨Windows、macOS、Linux平台运行，无平台依赖；
2.  GUI版本：兼容JavaFX 11及以上版本，适配主流操作系统的桌面分辨率。

### 7.3 可靠性需求
1.  系统运行无崩溃、无死循环，异常场景100%有捕获、有提示；
2.  数据持久化可靠性：地图数据修改后，实时同步至本地文件，断电、重启不丢失数据；
3.  容错性：用户输入错误数据时，系统可正常处理，不影响核心功能运行。

### 7.4 可扩展性需求
1.  算法可扩展：基于接口化设计，新增路径规划算法时，无需修改原有业务代码，符合开闭原则；
2.  地图数据可扩展：支持无上限新增地点、道路，适配不同规模的校园场景；
3.  功能可扩展：模块化设计，各模块之间低耦合，新增功能时不影响原有核心逻辑。

### 7.5 易用性需求
1.  操作流程极简，核心功能无学习成本，用户无需阅读说明书即可完成路径查询；
2.  菜单层级清晰，操作提示明确，无歧义；
3.  导航指引通俗易懂，符合用户日常步行导航的使用习惯。

## 八、核心数据结构设计
### 8.1 核心数据实体
1.  **地点实体（Vertex）**
    | 字段名 | 字段类型 | 约束 | 字段说明 |
    |--------|----------|------|----------|
    | id | String | 非空、唯一 | 地点唯一编号 |
    | name | String | 非空 | 地点名称 |
    | type | String | 非空 | 地点类型（教学楼/宿舍/食堂等） |
    | x | double | 非空 | 地点X轴坐标 |
    | y | double | 非空 | 地点Y轴坐标 |
    | description | String | 可选 | 地点备注说明 |

2.  **道路实体（Edge）**
    | 字段名 | 字段类型 | 约束 | 字段说明 |
    |--------|----------|------|----------|
    | fromVertex | Vertex | 非空 | 道路起点 |
    | toVertex | Vertex | 非空 | 道路终点 |
    | weight | double | 非空、>0 | 道路权重（距离，单位：米） |
    | isOneWay | boolean | 非空 | 是否单行道：true=单向，false=双向 |
    | isForbidden | boolean | 非空 | 是否禁行：true=禁行，false=可通行 |
    | roadType | String | 非空 | 道路类型（主路/小路/楼梯） |

3.  **路径结果实体（PathResult）**
    | 字段名 | 字段类型 | 字段说明 |
    |--------|----------|----------|
    | startVertex | Vertex | 路径起点 |
    | endVertex | Vertex | 路径终点 |
    | pathList | List<Vertex> | 途经点有序列表 |
    | totalDistance | double | 路径总距离 |
    | estimatedTime | double | 预计步行耗时（分钟） |
    | naviInstructions | List<String> | 分步导航指引列表 |

4.  **用户实体（User）**
    | 字段名 | 字段类型 | 约束 | 字段说明 |
    |--------|----------|------|----------|
    | username | String | 非空、唯一 | 账号 |
    | password | String | 非空 | 密码（加密存储） |
    | role | String | 非空 | 角色：admin=管理员，user=普通用户 |

### 8.2 数据持久化设计
1.  存储格式：采用JSON格式作为本地持久化格式，可读性强、解析方便，适配Java第三方JSON库（FastJSON/Jackson），也可简化为TXT/CSV格式（无第三方依赖）；
2.  存储文件分类：
    - `vertex.json`：存储全量地点数据；
    - `edge.json`：存储全量道路数据；
    - `user.json`：存储管理员账号信息；
    - `log.txt`：存储系统操作日志。

## 十、验收标准
### 10.1 功能验收标准
1.  地图数据管理功能：地点、道路的增删改查操作正常，数据可正确保存至本地文件，重启后可正常加载；
2.  路径规划功能：起点终点输入合法时，可正确计算最短路径，路径结果与人工计算结果一致，准确率100%；
3.  禁行绕行功能：设置禁行路段后，路径规划可自动绕行，不经过禁行路段；
4.  导航指引功能：可基于路径生成正确的分步导航指引，无逻辑错误、无方向歧义；
5.  权限控制功能：普通用户无法访问管理员功能，管理员需登录后才可操作地图管理功能，权限隔离正常；
6.  异常处理功能：所有非法输入、不可达路径、文件加载失败等异常场景，均有友好提示，系统无崩溃、无死循环。

### 10.2 非功能验收标准
1.  性能：单条路径计算响应时间≤1s，系统启动加载时间≤2s；
2.  兼容性：可在JDK8及以上版本正常运行，无版本兼容问题；
3.  代码规范：严格遵循Java面向对象设计思想，封装、继承、多态、接口特性体现完整，代码模块化、注释清晰、命名规范；
4.  易用性：菜单清晰、操作提示明确，核心功能3步内可完成，无学习成本。

## 十一、风险与应对方案
| 风险类型 | 风险描述 | 应对方案 |
|----------|----------|----------|
| 算法风险 | Dijkstra算法实现逻辑错误，导致路径计算结果不准确 | 1. 先完成小范围地图数据的算法测试，用人工计算结果校验；2. 采用优先队列优化算法，降低时间复杂度，避免大数据量下的性能问题 |
| 数据风险 | 地图数据量大，手动录入繁琐，易出错 | 1. 预设一套完整的校园地图默认数据，开箱即用；2. 开发地图数据批量导入功能，支持Excel/CSV文件批量录入 |
| 扩展风险 | 代码耦合度高，新增算法、功能时需大量修改原有代码 | 1. 严格遵循面向对象设计原则，基于接口实现业务逻辑，上层模块依赖抽象，不依赖具体实现；2. 做好模块拆分，各模块职责单一，低耦合高内聚 |
| 兼容性风险 | 开发环境与运行环境JDK版本不一致，导致无法运行 | 1. 统一基于JDK8版本开发，不使用高版本专属特性；2. 文档中明确标注运行环境要求，提供环境配置说明 |
